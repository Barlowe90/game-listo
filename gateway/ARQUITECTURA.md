### 1. Login (Ruta PГәblica)

```

        /v1/usuarios/auth/login
        
        вң“ Sin validaciГіn JWT (ruta pГәblica)

  вҖў Access Token (JWT, 15 min)

```

### 2. PeticiГіn Protegida

```
Cliente вҶ’ Gateway
        /v1/usuarios/auth/me
        Authorization: Bearer eyJ...
        
Gateway valida:
  вң“ Firma del token
  вң“ ExpiraciГіn
  вң“ No estГЎ revocado
  
Gateway agrega headers:

  X-User-Email: john@example.com
  X-User-Roles: USER,ADMIN
  
Gateway вҶ’ usuarios
        /v1/usuarios/auth/me
        + Headers X-User-*
        
usuarios:
  вҖў Lee X-User-Id del header
  вҖў NO necesita validar JWT (ya lo hizo Gateway)
  вҖў ConfГӯa en los headers
  
Cliente вҶҗ Gateway вҶҗ usuarios
        { "id": "123e4567...", "username": "johndoe", ... }
```

### 3. Logout (RevocaciГіn)

```
Cliente вҶ’ Gateway вҶ’ usuarios
        /v1/usuarios/auth/logout
        Authorization: Bearer eyJ...
        
usuarios:

PrГіxima peticiГіn con ese token:
  Gateway вҶ’ Redis вҶ’ Вҝexists revoked:jti:abc123...?

```

## рҹҺҜ Puntos Clave

1. **Stateless**: El Gateway no mantiene sesiones
2. **Secreto compartido**: `jwt.secret` debe ser idГ©ntico en Gateway y usuarios
3. **Fail-open**: Si Redis falla, el rate limiting permite peticiones (prioriza disponibilidad)
4. **Headers enriquecidos**: Los microservicios confГӯan en `X-User-*` (red interna segura)
5. **Orden de filtros**: RateLimitFilter (-50) вҶ’ JwtAuthenticationFilter (-100)
6. **TTL en Redis**: Los tokens revocados se limpian automГЎticamente cuando expiran

## рҹҡҖ Ventajas de Esta Arquitectura

вң… **CentralizaciГіn**: Un solo punto de autenticaciГіn  
вң… **Performance**: Microservicios no validan JWT (ya lo hizo Gateway)  
вң… **Seguridad**: RevocaciГіn inmediata con Redis  
вң… **Escalabilidad**: Gateway stateless puede replicarse  
вң… **Observabilidad**: Todo el trГЎfico pasa por un punto  
вң… **Rate limiting**: ProtecciГіn DDoS a nivel de infraestructura
